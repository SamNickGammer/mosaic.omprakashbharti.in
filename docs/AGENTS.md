## What is Mosaic?

Mosaic is a multi-client, multi-agent AI orchestration workspace built with **Next.js 14 (App Router)**. It lets a developer manage multiple clients (e.g. PepsiCo, SpinQuest, Bitcs), each with their own projects and AI agent credentials, from one interface.

The core feature: when a user sends a task, a **primary AI agent** analyzes it, **secondary agents** review and critique the response, and the primary agent produces one **final merged answer**. All agents are connected via their API keys stored encrypted in Neon DB.

## Key Rules — Always Follow

1. **App Router only** — never create files under `pages/`. Use `app/` exclusively.
2. **Server components by default** — only add `"use client"` when you actually need browser APIs or React state/hooks.
3. **Drizzle for all DB access** — no raw SQL strings. Schema lives in `lib/db/schema.ts`.
4. **Never log API keys** — not in console, not in error messages, not in traces.
5. **Encrypt at write, decrypt at read** — all `api_key_encrypted` fields use `lib/encryption.ts`.
6. **User isolation** — every DB query must filter by `userId` from the session. Never trust client-provided IDs without verification.
7. **Streaming via SSE** — agent chat responses must stream. Use `ReadableStream` + `text/event-stream` in route handlers.
8. **TypeScript strict mode** — no `any`, no `@ts-ignore` without a comment explaining why.
9. **Tailwind only for styling** — no CSS modules, no inline styles except for dynamic values.
10. **shadcn/ui for all UI components** — don't build from scratch what shadcn already has.

## Database Schema (Drizzle)

All tables are in `lib/db/schema.ts`. Key relationships:

```
users → clients → projects → tasks → task_messages
users → agent_accounts
projects ↔ agent_accounts (via project_agents join table)
task_messages → orchestration_traces
```

When adding a new table: update `schema.ts`, run `npx drizzle-kit generate`, commit the migration.

## Orchestration Logic (`lib/orchestrator/index.ts`)

The orchestrator follows this exact sequence for every task chat message:

```
Step 1: PRIMARY ANALYSIS
  → Call primary agent with: system prompt + project context + task instructions + user message
  → Stream partial tokens to client (SSE event: type="primary_stream")

Step 2: SECONDARY REVIEW (parallel)
  → For each secondary agent, call with:
     "Original task: [task]. Primary agent responded: [primary response].
      Review this response. Add anything missing, flag errors, suggest improvements.
      Be concise — your feedback will be used to improve the final answer."
  → Do NOT stream secondary responses to client (internal only)
  → SSE event: type="secondary_thinking" with agent name

Step 3: FINAL MERGE
  → Call primary agent again with:
     "Your initial response: [response].
      Feedback from other agents: [all secondary responses].
      Produce your final, improved answer. Only include what's actually useful from the feedback."
  → Stream final response to client (SSE event: type="final_stream")

Step 4: SAVE
  → Save full trace to orchestration_traces
  → Save final message to task_messages
```

SSE event format:
```json
{ "type": "primary_stream", "content": "..." }
{ "type": "secondary_thinking", "agentName": "Codex" }
{ "type": "final_stream", "content": "..." }
{ "type": "done", "messageId": "uuid" }
{ "type": "error", "message": "..." }
```

## What NOT to do

- Don't use `pages/` directory
- Don't store raw API keys anywhere — always encrypt before saving
- Don't call agents directly from components — always go through `/api/tasks/[id]/chat`
- Don't skip user ID checks in API routes
- Don't use `fetch` without error handling in route handlers
- Don't add new dependencies without checking if Vercel AI SDK already covers it
